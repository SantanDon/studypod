STUDYPODLM: YOUTUBE EXTRACTION PIPELINE FIXES

1. CLIENT-SIDE HOOK PERSISTENCE PATCH
   - Files Modified:
     * src/hooks/useYoutubeProcessing.tsx
     * src/hooks/useWebsiteProcessing.tsx
   - Problem: The hooks did not pass the target `notebookId` to `processDocumentAsync`, causing the system to fallback to updating local storage instead of writing the newly extracted chunks/embeddings to the backend database.
   - Solution: Added `notebookId` to the arguments passed to `processDocumentAsync` within both hooks, guaranteeing clean cloud persistence.

2. COOKIE PARSING RESILIENCY
   - File Modified: backend/src/routes/youtube.js
   - Problem: `response.headers.getSetCookie` is not universally supported across all Node.js runtimes. This caused session cookies to be empty, hindering InnerTube authorization.
   - Solution: Enhanced cookie retrieval by implementing a prioritized sequence: native `getSetCookie()`, checking raw headers, and splitting the standard `set-cookie` header array, ensuring cookies are robustly captured.

3. DIRECT INNERTUBE MWEB PLAYER BYPASS (HTTP 429 ELIMINATION)
   - File Modified: backend/src/routes/youtube.js
   - Problem: Sequence fetching of the watch page in a loop triggered YouTube's HTTP 429 (Too Many Requests) rate limit, rendering subsequent extraction attempts broken.
   - Solution: Implemented a two-tiered extraction strategy:
     * Tier 1 (Direct Fetch): Make an initial direct call to the `/youtubei/v1/player` InnerTube API using static mobile web parameters (MWEB client key and version) and without cookies. This successfully retrieves playability metadata and English caption tracks without hitting watch page rate limits.
     * Tier 2 (Scraping Fallback): If the direct MWEB player API call fails (due to key changes or custom restrictions), the parser falls back to fetching the watch HTML page, extracting the dynamic key/version/cookies, and retrying.
